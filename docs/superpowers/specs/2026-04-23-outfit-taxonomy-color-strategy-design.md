# OOTODAY 穿搭分类与配色策略设计

## 目标

把当前已上线的衣橱分类与 Today 推荐逻辑，从“能用的基础 taxonomy”升级为“可解释、可扩展、可产品化”的穿搭规则层。

这份设计主要解决 4 件事：

1. 给衣物分类一个更稳定的产品语义层，避免只停留在上传识图标签
2. 给颜色分类补上可用于推荐的结构，而不是只有展示用颜色名
3. 把内容平台里反复验证过的配色方法，收敛成 OOTODAY 可执行的规则
4. 为 Today、Inspiration、Shop、Travel 复用同一套穿搭判断语言

## 背景

2026-04-23 的外部调研中，可稳定验证热度的样本主要来自 YouTube 与 B 站，小红书和 X 以趋势补充为主。

高热内容虽然标题不同，但反复收敛到几条相同结论：

- 高级感通常不是“颜色稀有”，而是“颜色关系稳定”
- 新手最稳的方法不是大面积撞色，而是基础色 + 同类色 + 小面积强调色
- 多数高赞内容都强调“先控制整体，再制造一个重点”
- 真正耐用的穿搭方法依赖 3 个变量：颜色、轮廓、材质
- 鞋、包、外套、下装之间的呼应，往往比单品本身更决定成套感

代表性样本包括：

- YouTube: `Full Guide: EXPENSIVE Color Combos (& MISTAKES to Avoid)`，约 109 万观看
- YouTube: `THE LAST FASHION COLOR THEORY VIDEO YOU'LL EVER WATCH`，约 40 万观看
- YouTube: `This will COMPLETELY change the way you wear color`，约 14 万观看
- Bilibili: `史上最简单的色彩穿搭公式，百搭，但≠随便搭`，搜索摘要约 122 万播放
- Bilibili: `世界上最舒服的穿搭高级感配色`，搜索摘要约 111 万播放
- Bilibili: `穿搭如何配色？没有高级的衣服颜色，只有高级的色彩搭配！`，搜索摘要约 50 万播放

这些结论与 OOTODAY 当前方向高度一致，因为 Today、Shop、Inspiration、Travel 本质上都在回答同一个问题：

“这件衣服 / 这套搭配为什么成立，以及下一步怎么穿更稳。”

## 设计原则

### 1. 先做稳定规则，再做个性化

MVP 阶段优先使用用户容易理解且能解释的规则，避免过早依赖黑盒分数。

### 2. 优先支持日常可执行搭配

不是追求时尚编辑式造型，而是追求用户“今天真的能穿出去”的成功率。

### 3. 颜色规则要服务场景

同一件衣服在通勤、约会、旅行、内容拍照里的优先级不同，所以配色不是绝对规则，而是带场景权重的规则。

### 4. 统一语言

上传分类、推荐解释、购买分析、灵感复刻、旅行打包，最终都要落到同一套字段体系里，避免每个页面各讲一套话。

## 范围

### 本次包含

- 服装分类升级方案
- 颜色分类升级方案
- 搭配策略规则集
- 面向产品文案的解释模板
- 面向工程实现的字段建议

### 本次不包含

- 训练 ML 模型
- 像素级颜色提取服务
- 用户个人季型 / 肤色系统
- 完整的趋势引擎或社交媒体实时抓取

当前仓库已具备主分类 / 子分类体系。本次不是推翻，而是在其上增加推荐逻辑需要的结构化标签。

### 一级分类

- 全身装
- 外套
- 套装
- 内搭
- 鞋履
- 包袋
- 配饰
- 家居服

### 二级分类

沿用当前 taxonomy 中已有的中文子分类，保持上传表单、存量数据与推荐逻辑兼容。

### 新增推荐维度

每件衣物建议补充以下结构：

- `silhouette`
  - `fitted`
  - `straight`
  - `relaxed`
  - `oversized`
  - `flared`
- `length_profile`
  - `cropped`
  - `regular`
  - `long`
- `visual_weight`
  - `light`
  - `mid`
  - `heavy`
- `material_profile`
  - `soft`
  - `crisp`
  - `structured`
  - `textured`
  - `shiny`
- `pattern_profile`
  - `solid`
  - `stripe`
  - `check`
  - `graphic`
  - `floral`

这些字段的意义不是为了做衣橱展示，而是为了让推荐规则知道：

- 这件单品视觉上轻还是重
- 它更适合做主角还是打底
- 它会不会和另一件衣服同时抢注意力

## 颜色分类方案

当前 `colorCategory` 已经能支撑基础使用，但还不够支持“为什么推荐这套”的解释和计算。

建议把颜色拆成 5 层。

### 1. 展示色名 `color_label`

继续保留中文用户可读值，例如：

- 白色
- 米白色
- 卡其色
- 牛仔蓝
- 酒红色

### 2. 色族 `color_family`

- neutral
- red
- orange
- yellow
- green
- blue
- purple
- pink
- metallic

这层已在仓库中存在，可以继续复用。

### 3. 明度 `color_depth`

- light
- mid
- dark

这层也已存在，可以继续复用。

### 4. 饱和度 `color_intensity`

建议新增：

- muted
- balanced
- vivid

原因是调研里大量“高级感”判断都不是靠色相，而是靠低饱和。

### 5. 颜色角色 `color_role`

建议新增：

- `base`
- `support`
- `accent`

这不是衣物天然属性，而是衣物在当前造型里的角色。黑裤子通常适合做 `base`，红包更可能做 `accent`。

## 搭配策略

### 策略 1：60 / 30 / 10

默认造型按以下比例组织：

- 60% 主色
- 30% 辅助色
- 10% 强调色

在 OOTODAY 中可简化为：

- 上下装或全身装决定主色
- 外套或鞋包决定辅助色
- 配饰、小面积内搭、包或鞋决定强调色

### 策略 2：同色系优先

当用户没有明确风格偏好时，优先输出：

- 同色相不同深浅
- 同色家族不同材质
- 中性色内部层次

原因：

- 最稳
- 最不挑人
- 与通勤、旅行、日常高频场景最兼容

### 策略 3：邻近色优先于撞色

推荐优先级应为：

1. 同色系
2. 邻近色
3. 中性色 + 单彩色
4. 小面积撞色

大面积双高饱和撞色不作为默认推荐，只在用户明确偏好个性化、拍照感、内容感时抬权。

### 策略 4：首尾呼应

当造型需要更强完成度时，优先检查：

- 上衣与鞋同家族
- 包与鞋同家族
- 外套与下装同深度

这是调研中非常高频、也最容易产品解释的一类规则。

### 策略 5：三明治配色

当上衣和鞋同色，中间下装作为过渡色时，整体很容易成立。

适合：

- Today 快速推荐
- 通勤场景
- 男装与中性风

### 策略 6：高饱和颜色只做一个重点

规则化表达：

- 当 `vivid` 单品数量大于 1 时降分
- 当 `vivid` 与 `vivid` 且非同家族同时出现时进一步降分
- 当 `vivid` 只出现于鞋 / 包 / 配饰时加分

### 策略 7：视觉重量平衡

除颜色外，还要看量感。

例如：

- 厚重外套 + 厚重鞋 + 深色下装，整体可能太压
- 轻薄上装 + 轻薄下装 + 轻色鞋，整体可能缺少落点

建议增加一条平衡规则：

- 造型里至少有一个 `visual_weight = heavy` 或 `dark` 的锚点
- 同时避免超过两个重锚点

### 策略 8：场景权重

同一规则在不同页面权重不同：

- Today：稳妥、好执行、天气兼容优先
- Shop：与现有衣橱的可搭套数优先
- Inspiration：风格相似度优先，允许更高对比
- Travel：可复穿率、少件多搭优先

## 推荐打分思路

建议在现有 Today 规则基础上新增以下分项：

- `base_color_match_score`
  - 是否存在稳定基础色锚点
- `color_harmony_score`
  - 同色系 / 邻近色 / 中性色组合是否成立
- `accent_control_score`
  - 是否只有一个重点色
- `weight_balance_score`
  - 视觉量感是否失衡
- `scene_fit_score`
  - 是否符合通勤 / 休闲 / 出行等场景目标
- `repeatability_score`
  - 这套是否容易复穿、是否值得在 Travel / Shop 中复用

## 解释文案模板

为了让规则结果能被用户理解，建议统一解释风格。

### Today

- “这套以 `米白 + 卡其` 做主色，整体稳定，鞋子再用同色系呼应，出门不容易出错。”
- “上身和彩色单品只保留了一处重点，所以看起来更利落，不会花。”

### Shop

- “这件单品能接入你现有的 `黑 / 米 / 牛仔蓝` 基础色体系，所以可搭次数高。”
- “颜色存在感较强，适合做点缀，不建议再和另一件高饱和单品同时购买。”

### Inspiration

- “原图成立的关键不是单件相同，而是 `低饱和中性色 + 深浅层次 + 小面积亮点`。”

### Travel

- “这几件围绕同一组基础色展开，所以 5 天行程里可以反复组合，不会每天都像新开一套箱子。”

## 落地建议

### Phase 1：文档对齐

- 先把这套结构作为 repo 内统一规格
- 不急着一次把所有字段写入数据库

### Phase 2：taxonomy 扩展

- 在 `lib/closet/taxonomy.ts` 补充 `color_intensity` 所需映射
- 先给高频颜色补上强弱标签
- 先通过规则映射生成，不要求 AI 一步识别完全准确

### Phase 3：Today 规则升级

- 在 `generate-recommendations` 中加入颜色角色、重点色数量、首尾呼应与量感平衡

### Phase 4：跨页面复用

- Shop 用同一套规则解释“值不值得买”
- Inspiration 用同一套规则解释“原图为什么成立”
- Travel 用同一套规则解释“少带几件也能搭”

## 风险与取舍

### 风险 1：字段太多，识图不稳定

处理方式：

- 把一部分字段定义为规则推导值，而不是识图直出值

### 风险 2：颜色命名过细，用户难以维护

处理方式：

- 用户界面保留简单中文色名
- 推荐层内部再映射到 family / depth / intensity

### 风险 3：规则过强，推荐显得保守

处理方式：

- 默认稳妥
- 用户明确偏好“更有存在感 / 更会拍 / 更个性”时再放开强调色阈值

## 验收标准

这份设计真正完成，不是文档写完，而是后续实现能做到：

1. 同一套衣物在 Today、Shop、Inspiration、Travel 中使用同一套颜色语言
2. 推荐理由能说清楚“为什么配得上”
3. 购买分析能说清楚“为什么值不值得买”
4. 旅行打包能说清楚“为什么这几件能反复穿”
5. taxonomy 的后续扩展不需要重做所有页面文案和规则
